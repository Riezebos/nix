# Architecture

This repository is a personal Nix flake with one rule above all others:
`flake.nix` stays boring. It delegates the real work to flake-parts modules under
`modules/`.

## Flake shape

`flake.nix` is just:

- `flake-parts.lib.mkFlake`
- `import-tree ./modules`

Because `import-tree` auto-loads every `.nix` file under `modules/`, every file
there must be a flake-parts module: a function accepting arguments such as
`{ self, inputs, lib, ... }`.

A plain NixOS module such as `{ config, pkgs, ... }: { ... }` will fail
confusingly if placed directly under `modules/`. Export plain modules from a
flake-parts wrapper instead:

```nix
{...}: {
  flake.nixosModules.example = {config, pkgs, ...}: {
    # real NixOS module body
  };
}
```

`hosts/foundry/hardware-configuration.nix` intentionally lives outside
`modules/` because it is generated by `nixos-generate-config` and is not a
flake-parts module.

## nixpkgs inputs

There are two nixpkgs roots on purpose.

| Input | Channel | Used for |
|---|---|---|
| `nixpkgs` | `nixos-25.11` | `nixosConfigurations.foundry` |
| `nixpkgs-devenv` | `cachix/devenv-nixpkgs/rolling` | Home Manager and nix-darwin |

`home-manager` and `darwin` follow `nixpkgs-devenv`. `disko`, `sops-nix`, and
`deploy-rs` follow stable `nixpkgs`.

The important invariant: rolling packages are allowed on laptops, not on the
server.

## Configurations

Home Manager:

- `simon-darwin`: aarch64, `/Users/simon`
- `simon-m4`: aarch64, `/Users/simon.riezebos`
- `simon-linux`: x86_64, `/home/simon`

All three import `flake.homeModules.shared` from `modules/home/shared.nix`.

nix-darwin:

- `Simons-MacBook-Air`, importing `flake.darwinModules.system`

NixOS:

- `foundry`, importing `flake.nixosModules.foundryConfiguration`

## Module map

```text
modules/
  parts.nix
  home/
    configurations.nix
    shared.nix
  darwin/
    configurations.nix
    system.nix
  hosts/foundry/
    default.nix
    configuration.nix
    disko.nix
    hardware.nix
    secrets.yaml
  features/
    alerting.nix
    authentik.nix
    caddy.nix
    crowdsec.nix
    foundryvtt.nix
    monitoring.nix
    postgresql.nix
    restic.nix
  deploy.nix
  pre-commit.nix
```

Feature modules export reusable NixOS modules through `flake.nixosModules.*`.
The host module decides which features are active on `foundry`.

## CI and pre-commit

Pre-commit hooks live in `modules/pre-commit.nix` through `git-hooks.nix`.

The active hooks are:

- `gitleaks`, scanning staged changes for secrets with redacted output
- `alejandra`, formatting Nix files

GitHub Actions builds the x86_64 Linux checks and the `foundry` system closure.
Deploys from `main` use deploy-rs and its rollback health check.

`nix flake check` is still the local all-up validation command, but CI is more
selective because the Linux runner cannot build the aarch64 Darwin Home Manager
inputs.

## Secrets and host identity

The public repo must not contain the real server IP. Local commands should use
`Host foundry` from `~/.ssh/config`.

Server secrets live in `modules/hosts/foundry/secrets.yaml` and are encrypted
with sops. The server decrypts them through its SSH host key converted to an age
recipient with `ssh-to-age`; the laptop's personal age recipient remains in
`.sops.yaml` for editing.
