## Common commands

```bash
# Apply home-manager config on the primary Mac
home-manager switch --flake .#simon-darwin
# (or use the shell alias: hm-mac)

# Validate the whole flake (also runs pre-commit hooks)
nix flake check

# Enter the dev shell (installs pre-commit hooks via direnv or explicit)
nix develop

# Format all Nix files
alejandra .

# Edit sops-encrypted secrets for the server
sops modules/hosts/foundry/secrets.yaml

# Deploy to server (run from laptop; NIX_SSHOPTS must point to deploy key)
export NIX_SSHOPTS="-o IdentitiesOnly=yes -i $HOME/.config/foundry-bootstrap/deploy_ed25519"
nix run nixpkgs#nixos-rebuild -- switch \
  --flake .#foundry \
  --target-host deploy@foundry \
  --build-host  deploy@foundry \
  --no-reexec \
  --sudo

# LUKS unlock after server reboot
foundry-unlock          # reads passphrase from macOS Keychain, SSHes to initrd on :2222
foundry-unlock-seed     # one-time: stores passphrase in Keychain
```

## Architecture

### Flake structure

`flake.nix` is intentionally minimal — just `flake-parts.lib.mkFlake` + `import-tree ./modules`. Every `.nix` file under `modules/` is **auto-loaded and must be a valid flake-parts module** (a function accepting `{ self, inputs, lib, ... }`). A plain NixOS module (`{ config, pkgs, ... }`) will eval-fail confusingly if dropped into `modules/` unwrapped.

`hosts/foundry/hardware-configuration.nix` lives *outside* `modules/` by design — it is a plain NixOS module generated by `nixos-generate-config` and import-tree would choke on it.

### Two separate nixpkgs inputs

| Input | Channel | Used for |
|---|---|---|
| `nixpkgs` | `nixos-25.11` (stable) | `nixosConfigurations.foundry` |
| `nixpkgs-devenv` | `cachix/devenv-nixpkgs/rolling` | all `homeConfigurations.*` and `darwinConfigurations.*` |

`home-manager` and `darwin` follow `nixpkgs-devenv`; `disko`, `sops-nix`, `deploy-rs` follow `nixpkgs`. Do not let rolling packages bleed onto the server.

### Configurations defined

- **Home Manager**: `simon-darwin` (aarch64, `/Users/simon`), `simon-m4` (aarch64, `/Users/simon.riezebos`), `simon-linux` (x86_64, `/home/simon`) — all share `flake.homeModules.shared` (`modules/home/shared.nix`)
- **nix-darwin**: `Simons-MacBook-Air` — imports `flake.darwinModules.system`
- **NixOS server**: `foundry` (x86_64-linux, Hetzner) — imports `flake.nixosModules.foundryConfiguration`

### Server (`foundry`) highlights

- **Disk layout**: mdraid RAID1 across two Samsung PM983 NVMes + LUKS2 on top + ext4. Separate mirrored `/boot` (unencrypted, GRUB-readable). Legacy BIOS, no TPM.
- **LUKS unlock**: SSH into initrd on port 2222 (`ForceCommand systemd-tty-ask-password-agent --query`). The `foundry-unlock` zsh helper (in `modules/home/shared.nix`) automates this from the laptop.
- **Secrets**: sops-nix, encrypted with age keys. The server decrypts using `/etc/ssh/ssh_host_ed25519_key` → age (via `ssh-to-age`). `modules/hosts/foundry/secrets.yaml` must be `git add`-ed before `nix eval` can read it (flakes filter through git's tree view).
- **Unattended upgrades**: `system.autoUpgrade` pulls from `github:Riezebos/nix` nightly at 04:30 with `operation = "boot"` — stages the generation without activating, so services don't restart mid-session.
- **Firewall**: explicit allow-list — ports 22, 2222 only. Port 80/443 will be added in Phase 4 (Caddy).
- **No public IP in this repo**: the real server IP lives in `~/.ssh/config` as `Host foundry`. The `foundry-unlock` helper resolves it at runtime via `ssh -G foundry`.

### Pre-commit hooks

Defined in `modules/pre-commit.nix` using `git-hooks.nix`. Uses `prek` (Rust re-implementation of pre-commit — ~75× faster):
- **gitleaks** — scans staged changes for secrets; `--redact` keeps findings out of terminal scrollback. Does **not** catch IPs/hostnames — that's a discipline requirement.
- **alejandra** — Nix formatter, excludes `hosts/foundry/hardware-configuration.nix` to avoid reformat/regen conflicts.

`nix flake check` re-runs the same hooks, so CI gets the identical gate.
